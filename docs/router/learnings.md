# Router Development Learnings

Field notes from building an intent-aware tool router for a Claude Code proxy, March 2026.

1. **Your first routing signal is probably garbage.** We fed the router the last user message for intent detection. Turns out Claude Code injects `<system-reminder>`, `<command-message>`, and session continuation boilerplate into user messages. The router was classifying "This session is being continued from a previous conversation..." against tool catalog embeddings and getting random noise back. Fix the input before tuning the model.

2. **"Include everything" is the wrong safe default for token-expensive tools.** Our initial fallback (`fallback_all`) included all 170 tools when neither rules nor embeddings matched. This felt safe but meant 44% of eligible requests produced zero savings — the router existed but did nothing. Inverting to "exclude specialized groups by default" (`default_core_only`) immediately moved savings from 40% to 83% of eligible requests.

3. **Shadow mode is not optional, it's the product.** We almost shipped auto-filtering after the first rules+embeddings prototype looked clean. Shadow mode caught that our miss rate was 5.7% against a 2% target. Both misses were the same pattern: user says something generic ("improve your strategy"), model autonomously decides to query Supabase. We would have broken real workflows if we'd skipped shadow.

4. **The hardest misses are unpredictable tool choices.** Message-level intent routing assumes the user's words predict which tools the model will use. This breaks when the model creatively reaches for a tool the user never mentioned. "Do we have data showing that?" doesn't say "Supabase" anywhere, but the model ran three SQL queries. No amount of keyword or embedding tuning fixes this class of miss — it needs session-level or model-in-the-loop signals.

5. **Evaluation that skips the interesting cases is worthless.** Our `computeEvaluation()` originally skipped `fallback_all` events — the exact events where the router wasn't making a decision. This meant `would_have_missed` was always 0%, which looked perfect but measured nothing. When we started evaluating `default_core_only` decisions, the miss rate jumped to 5.7%. The metric was lying by omission.

6. **Tag stripping needs to be consistent across all text paths.** We fixed infrastructure tag stripping in the router's intent extraction but forgot the grouping helpers (`extractFirstHumanText` / `extractLastHumanText`). Result: the router saw clean text while the conversation grouper saw `<command-message>plan-ceo-review</command-message>` as a new conversation boundary, creating bogus turn groups. One shared `stripInfrastructureTags()` function, three call sites — obvious in hindsight.

7. **Your UI will lie to you about shadow mode.** The detail panel showed "Stripped groups: context7, firebase, linear, playwright, supabase" while all 170 tools were forwarded. Users (including us) read "stripped" as "removed." Changing labels to "Would strip" / "Would keep" / "Potential savings" in shadow mode eliminated the confusion. Hypothetical predictions need hypothetical language.

8. **Small sample sizes hide real problems.** 35 post-change events showed 2 misses (5.7%). Is that a real 5% miss rate or bad luck in a small sample? We don't know yet. The spec says 200+ events minimum before graduating to auto mode. Resist the urge to ship based on "it looks fine so far" — the first 35 events looked fine for `fallback_all` too, because we weren't measuring.

9. **Rules will match your own UI.** When a user views the Jannal dashboard via Playwright, the rendered text ("linear, firebase, playwright, context7, supabase") enters the conversation. Our keyword rules dutifully matched every group name, selected everything, and produced 0 savings. Self-referential content is a real input class. Not worth fixing (it's harmless), but worth knowing about.

10. **The `stripEligible` flag is a meaningful safety boundary.** Instead of "strip all catalog groups when there's no signal," we mark each group explicitly as `stripEligible: true`. Today all five are strip-eligible, but the distinction matters: a future catalog entry (say, a code generation MCP server) might not be safe to default-strip. Making the safety boundary explicit in the catalog prevents implicit assumptions from becoming silent regressions.

11. **Prompt caching changes your cost model completely.** Early token/cost displays showed "Actual input: 1 token" for requests with 1.4M estimated tokens. The Anthropic API splits input into `input_tokens` (uncached), `cache_creation_input_tokens`, and `cache_read_input_tokens`. We were only reading the first field. With prompt caching, a 1.4M token request might have 1.4M cache reads at 10% rate — the real cost is $0.75, not the $7.23 we were showing. Cache-aware pricing isn't a nice-to-have; it's table stakes for a proxy tool.

12. **The polarity of your default determines your product.** `fallback_all` (include everything when unsure) makes the router an opt-in optimizer: it only saves tokens when it's confident. `default_core_only` (exclude specialized when unsure) makes it an opt-out safety net: it always saves tokens unless there's a reason not to. Same signals, same confidence thresholds, completely different product behavior. We spent days tuning signals when the real lever was the one-line default branch.
